# opentui-island agent notes

## purpose

- Embed OpenTUI islands inside Node terminal UIs such as `pi-tui` and Ink.
- Keep the host in Node and render the island in a local Bun sidecar.
- Favor small, reusable host-side APIs over app-specific helpers.

## entry points

- Public exports: `src/index.ts`
- Low-level host bridge: `src/sidecar/client.ts`
- Sidecar server: `src/sidecar/server.ts`
- Shared bridge types/hooks: `src/core/bridge.tsx`
- `pi-tui` adapter: `src/adapters/pi-tui/index.ts`
- Ink adapter: `src/adapters/ink/index.tsx`
- Examples: `examples/`
- Integration coverage: `test/`

## architecture

- Host apps run in Node.
- OpenTUI islands run in a Bun sidecar process.
- All host runtimes share the same normalized bridge and sidecar protocol.
- Prefer one implementation path per capability instead of adapter-specific duplicates.
- Keep result/event flows generic to the library rather than naming them after one app.

## working rules

- Prefer minimal API additions over helper proliferation.
- Keep `pi-tui` and Ink examples consumer-facing and package-oriented.
- When changing bridge behavior, update both README examples and integration tests.
- Do not add app-specific language to the generic bridge examples.
- Preserve Bun as the repo runtime and package manager unless the user asks otherwise.

## validation

- `bun run check`
- `bun test`
- `bun run test:node-integration`
- `OPENTUI_TTY_SMOKE_INK=0 bun run test:tty-smoke`
- `bun run check:pack`

## changelog and releases

- Maintain top-level `CHANGELOG.md` as the source of truth for user-visible changes.
- Keep upcoming work under `## [Unreleased]` with `### Added`, `### Changed`, and `### Fixed`.
- Append to existing subsections instead of creating duplicates.
- When cutting a release, move unreleased entries into a versioned section and start a fresh `## [Unreleased]` section.
- Use changelog entries as the basis for GitHub release notes instead of trusting autogenerated notes blindly.

## commits

- Commit titles should follow Conventional Commits: `<type>[scope]: <description>`.
- Prefer descriptions that explain why the change matters to consumers or maintainers.
